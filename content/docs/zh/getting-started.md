# 开始使用

Previously 用一条垂直时间轴上的单一、持续的关系取代了聊天线程——你只需出现并交谈，而 Agent 会自行决定每条消息需要哪些过往上下文。

## 界面

固定的 **AppHeader** 横跨每个页面的顶部：左侧是"Previously"标志，右侧依次是 GitHub、文档、设置、主题切换和语言切换。无论滚动位置如何，始终可见。

页面是一个垂直滚动区域，分为三个部分，从上到下：

1. **全屏 Hero** —— `"Previously on {你的名字}"` 以电影级文字渐显动画呈现。导航链接和外观控件已移至 AppHeader，保持 Hero 区域简洁专注。
2. **记忆时间轴** —— 你过去的 slices，最早的在顶部，最新的在底部，按日期分组，带有每个 slice 的时间戳，以及用于加载更早历史的"加载更多"按钮。
3. **实时聊天消息** —— 正在进行中的对话，实时流式输出。

在这三个部分下方，**固定在屏幕底部**，是一个粘性输入栏：一个文本输入框，可自动调整大小，从 24 px 到最大 72 px。回车发送；Shift+Enter 换行。当 Agent 正在流式输出时，发送箭头变为方形停止按钮。工具栏提供图片附件功能（粘贴、拖放或文件选择器；仅限 `image/*`，每文件 10 MB）。

整个布局位于一个 `MessageScroller` 内部——向上滚动回顾过去，向下滚动继续上次的对话。

> **关键要点：没有会话列表，没有线程管理，没有"新建聊天"按钮。时间轴本身就是会话列表。**

## 记忆如何在你对话中形成

记忆不是你手动管理的东西——它在你的对话中自动形成。

**Slice** 是一个对话片段。当你开始输入时开启，在你活跃期间保持开启，并在静默 30 分钟后关闭。这是唯一的规则。没有容量限制，没有话题切换检测，没有基于机器学习的错误分割。纯粹的时间驱动切片。

每个关闭的 slice 会写入一个 Markdown 文件，路径为：

```
memory/episodic/slices/2025/11/21/0825.md
```

路径即为时间戳：年、月、日以及第一条消息的时-分。文件带有 YAML 前置元数据——关注点、摘要、决策、未决事项、情绪基调以及标签——全部由 Agent 在每轮对话中维护。你不需要手动标注任何内容。

按月按年从上到下阅读，slice 目录就是你的自传。可 Git 差异对比（git-diffable）、人类可读、可移植。

## 询问过去：观察召回过程

当你问及涉及过去的问题——比如"我对那个房产项目做了什么决定？"或"提醒我上次和 Sarah 聊天是什么时候"——Previously 会运行一个可见的两层召回机制，结果以可展开卡片的形式呈现在答案上方。

1. **Flash 召回**（约 500 毫秒，DeepSeek-chat）——扫描最近的 slice 摘要和 strand 索引（`memory/episodic/strands.json`）。返回指向相关 slice 的**指针**：slice ID、相关性评分以及每条命中的理由。这是快速、近似的扫描。同时作为副作用维护当前活跃 slice 的元数据（摘要、标签、未决事项）——无需额外的后处理步骤。

2. **深度召回**（Pro，主模型）——接收 Flash 的指针，决定通过 `readMemory` 工具完整读取哪些 slice。工具调用显示为内联卡片，标签为 `"Recalling in detail..."`。如果 Flash 没有找到任何内容，模型可以自行探索目录。

召回结果在共享的 `ToolLayout` 卡片内渲染，带有历史图标：

- **折叠状态**：显示经过的时间（例如 `"1s"`）。
- **展开状态**：显示 Flash 的推理文字、召回命中列表（slice ID、理由、相关性百分比）以及它认为相关的标签。

深度召回期间的工具调用渲染为内联卡片，带有友好的人类可读标签：

| 工具 | 你看到的文字 |
|------|------------|
| `readMemory` | Recalling in detail... |
| `listMemory` | Recalling more... |
| `readIndex` | Scanning timeline... |

每张卡片有五种视觉状态：**运行中**（旋转加载器）、**已完成**（绿点）、**错误**（红色）、**中断**（黄色）、**被拒绝**（红色）。点击或按 Enter/Space 展开以查看完整的工具输出——无需 JS 高度测量，仅使用 CSS `grid-template-rows` 过渡。

消息本身分三个阶段渲染，与 Agent 的内部执行顺序一致：**Recall** → **Reasoning** → **Response**。你看到 Agent 扫描时间轴、思考、然后回答——全部在一条消息内。

## 首次体验：演示模式

Previously 没有内置的引导向导。取而代之，它提供了一个 **DEMO_MODE** 环境标志，将部署转变为只读导览。

在环境中设置 `DEMO_MODE=true`，则所有对 `memory/` 的读取都将重定向到 `memory/demo/personal_14/`——一个预置的虚构角色数据集，包含 30+ 个 slice 和跨越数年的对话。屏幕顶部有一个固定横幅，提示你写入的任何内容都不会被保存。写入操作看似成功，但永远不会到达仓库；刷新页面后一切重置。

在线演示位于 [previously-demo.ldwid.com](https://previously-demo.ldwid.com)。它运行完整的两层召回系统（Flash + Pro），因此你可以浏览时间轴、翻阅过去的 slice、自由聊天并观察召回过程——无需自己部署。

> **重要提示：演示数据集不在仓库的 `main` 分支上。** 它位于 `DEMO_REF` 分支上，当 `DEMO_MODE` 开启时，通过绑定到该分支的 GitHub 读取来提供服务。如果在本地 `main` 分支上尝试浏览 `memory/demo/personal_14/`，将找不到任何内容。

### 启动你自己的时间轴

要从演示模式切换到自己的时间轴：

1. 使用你自己的 GitHub 仓库和 GitHub 个人访问令牌（contents 读写权限）部署 Previously。
2. 不设置 `DEMO_MODE` 或将其设为 `false`。
3. 就这样。你输入的第一条消息就会开启你的第一个 slice。

无需创建账号，无需设置数据库，无需种子数据。时间轴从空白开始。记忆从你的第一次对话开始逐步累积。

## Strands：跨越时间的语义层

随着 slice 不断累积，Agent 会构建一个轻量级的关键词索引，称为 **strands**。Strand 是一个跨多个 slice 重复出现的话题关键词——比如 `work`、`family`、`housing`。文件 `memory/episodic/strands.json` 将每个 strand 映射到所有包含该关键词的 slice。

Flash 在召回过程中扫描 strand 索引，以按话题查找 slice，而不仅仅按时间远近。如果你在三年间跨越 30 个 slice 都聊过"housing"，Flash 通过 strand 索引找到它们，然后 Pro 读取实际内容。

```preview
demo: strands-index
```

> **路线图：** 当前的 strands 是一个轻量的关键词到 slice 的索引。未来里程碑会演进为更完善的一等公民 strand，带有滚动摘要和深度召回整合。目前，该索引是无损的——它指向每个相关的 slice，由 Pro 读取所需内容。

## 日常使用意味着什么

不需要任何仪式。你不需要打开"新的对话"。不需要打标签或分类。不需要管理记忆。

- **出现** —— 打开页面。时间轴已经加载好，带着你最近的过往。Hero 显示 "Previously on {你的名字}"。
- **交谈** —— 输入并按下回车。如果已过去 30 分钟以上，会开启一个新的 slice。Agent 读取时间轴，扫描相关上下文，然后回应。
- **询问过去** —— "关于 X 我做了什么决定？"或"Y 是什么时候发生的？"Flash 扫描摘要和 strand 索引，Pro 读取完整的 slice，你可以看到工具卡片出现并逐步解析完成。
- **滚动** —— 实时聊天上方的时间轴就是你的完整历史。按日期分组、分页，带有电影风格的"N 分钟/小时/天之后"标题卡，填补最后记录时刻与现在之间的空白。

这就是整个循环。无需设置，无需清理，无需线程管理。只有一个持续的对话，带着真正工作的记忆。

## 项目状态

**Previously 处于活跃的早期开发阶段，尚未准备好用于个人或生产环境**（状态：实验性）。本指南记录的是代码中当前存在的 v0.1.0 版本。核心架构已可运行，但许多子系统仍在设计和构建中。本"开始使用"文档以个人自托管部署为背景，而非 SaaS 注册。
